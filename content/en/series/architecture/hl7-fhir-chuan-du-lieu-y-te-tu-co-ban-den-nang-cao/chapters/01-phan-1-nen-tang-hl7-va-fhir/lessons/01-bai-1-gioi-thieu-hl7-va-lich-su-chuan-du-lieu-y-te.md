<h2 id="1-tai-sao-can-chuan-du-lieu-y-te"><strong>1. Why is it necessary to standardize medical data?</strong></h2>

<p>Imagine you go to hospital A for a check-up, are diagnosed with type 2 diabetes and prescribed medication. Next week, you go to hospital B for another reason. Doctors at hospital B cannot see your medical records at hospital A — because the two systems cannot "talk" to each other at all.</p>

<p>This is not a rare story. Actually, this is <strong>most common problem</strong> in the digital health industry globally. Each hospital and each clinic uses different software, stores data in different ways, and does not have a "common language" to exchange information.</p>

<h3 id="interoperability-la-gi"><strong>What is interoperability?</strong></h3>

<p><strong>Interoperability</strong> (interoperability) in healthcare is the ability of different healthcare information systems to:</p>

<ul>
<li><strong>Data exchange</strong> (Exchange) — send and receive data between systems</li>
<li><strong>Understand the data</strong> (Interpret) — the receiving system can understand the correct meaning of the data</li>
<li><strong>Data usage</strong> (Use) — the data received can be used to support clinical decision making</li>
</ul>

<p>There are 4 levels of Interoperability:</p>

<table>
<thead>
<tr><th>Level</th><th>Name</th><th>Description</th><th>For example</th></tr>
</thead>
<tbody>
<tr><td>1</td><td>Foundational</td><td>Send/receive data between 2 systems</td><td>Send PDF file of test results</td></tr>
<tr><td>2</td><td>Structural</td><td>Data has a uniform structure</td><td>Send test results as HL7 v2 message</td></tr>
<tr><td>3</td><td>Semantics</td><td>Both sides understand the same meaning</td><td>ICD-10 code "E11.9" is understood as "Type 2 diabetes mellitus"</td></tr>
<tr><td>4</td><td>Organizational</td><td>There are processes, policies, and legal support</td><td>The circular allows data sharing between hospitals</td></tr>
</tbody>
</table>

<h3 id="hau-qua-thieu-interoperability"><strong>Consequences of Lack of Interoperability</strong></h3>

<ul>
<li><p><strong>Repeat testing</strong> — the patient had to retake the test because the new hospital did not have the old results</p></li>
<li><p><strong>Medical errors</strong> — The doctor does not know what medications the patient is allergic to or what medications he is taking</p></li>
<li><p><strong>Costs increase</strong> — it is estimated that the US wastes $30 billion/year due to lack of interoperability</p></li>
<li><p><strong>Delay treatment</strong> — have to wait for paper transfer documents</p></li>
<li><p><strong>Medical research is limited</strong> — multicenter data cannot be aggregated</p></li>
</ul>

<h2 id="2-hl7-international"><strong>2. HL7 International — The organization behind the medical data standard</strong></h2>

<p><strong>HL7 (Health Level Seven)</strong> International is a non-profit standards organization founded in <strong>1987</strong>, headquartered in Ann Arbor, Michigan, USA. The name "Level Seven" refers to the 7th layer (Application Layer) in the OSI model — the layer where applications communicate with each other.</p>

<p>HL7 International has more <strong>1,600 members</strong> more words <strong>55 countries</strong>, including medical software vendors, hospitals, government organizations, insurance carriers, and research organizations.</p>

<h3 id="cac-chuan-hl7-da-phat-trien"><strong>HL7 standards have evolved</strong></h3>

<p>Over more than 35 years, HL7 has developed many data standards, each addressing the needs of the times:</p>

<h2 id="3-hl7-v2"><strong>3. HL7 Version 2 (v2) — The world's most popular standard</strong></h2>

<h3 id="lich-su-hl7-v2"><strong>History</strong></h3>

<p>HL7 v2 was released first this year <strong>1989</strong> and quickly became the world's most popular medical data exchange standard. To date, est <strong>95% of hospitals in the US</strong> and <strong>35+ countries</strong> Use HL7 v2.</p>

<h3 id="cau-truc-hl7-v2"><strong>HL7 v2 Message structure</strong></h3>

<p>HL7 v2 uses a text format with pipe-delimited characters:</p>

<pre><code>MSH|^~\&amp;|HIS|BVBACHMAI|LIS|LABXN|202603301000||ADT^A01|MSG00001|P|2.5
EVN|A01|202603301000
PID|1||MRN12345^^^BVBACHMAI||NGUYEN^VAN^A||19850315|M|||123 Le Loi^^HCM^^700000^VN
PV1|1|I|W4B^401^1|||||||||||||||VN001|||||||||||||||||||||||||202603300800
</code></pre>

<p>Explanation:</p>
<ul>
<li><p><strong>MSH</strong> — Message Header: information about the message (source, destination, type, version)</p></li>
<li><p><strong>EVN</strong> — Event: event that triggers the message (A01 = hospitalization)</p></li>
<li><p><strong>PID</strong> — Patient Identification: patient information</p></li>
<li><p><strong>PV1</strong> — Patient Visit: information about visits/hospitalizations</p></li>
</ul>

<h3 id="uu-nhuoc-diem-hl7-v2"><strong>Advantages and disadvantages</strong></h3>

<table>
<thead>
<tr><th>Advantages</th><th>Disadvantages</th></tr>
</thead>
<tbody>
<tr><td>Widely popular, well supported</td><td>Too many options, each implementation is different</td></tr>
<tr><td>Simple, lightweight</td><td>There is no strict model (each field can be used differently)</td></tr>
<tr><td>Millions of interfaces running</td><td>Backward compatibility complex (v2.1 → v2.9)</td></tr>
<tr><td>Many support tools (Mirth, Rhapsody)</td><td>Does not support web/REST natively</td></tr>
</tbody>
</table>

<h2 id="4-hl7-v3-rim"><strong>4. HL7 Version 3 and Reference Information Model (RIM)</strong></h2>

<h3 id="tham-vong-v3"><strong>Ambition for radical standardization</strong></h3>

<p>Realizing the limitations of v2, HL7 began to develop <strong>v3</strong> since the late 1990s with the ambition to create a unified, coherent data model for the entire healthcare sector.</p>

<p>At the heart of HL7 v3 is <strong>RIM (Reference Information Model)</strong> — an abstract object model that describes all concepts in healthcare:</p>

<ul>
<li><p><strong>Act</strong> — medical actions (examination, testing, prescription...)</p></li>
<li><p><strong>Entity</strong> — entity (patient, doctor, drug, device...)</p></li>
<li><p><strong>Role</strong> — role (patient, healthcare worker, provider...)</p></li>
<li><p><strong>Participation</strong> — participate (who participates in which action)</p></li>
<li><p><strong>ActRelationship</strong> — relationships between actions</p></li>
<li><p><strong>RoleLink</strong> — relationships between roles</p></li>
</ul>

<h3 id="van-de-v3"><strong>Issues with HL7 v3</strong></h3>

<p>Although v3/RIM is very tight in theory, in practice:</p>

<ul>
<li><p><strong>Too complicated</strong> — XML messages are cumbersome and difficult to implement</p></li>
<li><p><strong>Difficult learning curve</strong> — need a deep understanding of RIM to be able to deploy</p></li>
<li><p><strong>High cost</strong> — the time and resources to implement are huge</p></li>
<li><p><strong>Low adoption</strong> — very few organizations have successfully deployed pure HL7 v3</p></li>
</ul>

<h2 id="5-cda"><strong>5. CDA (Clinical Document Architecture)</strong></h2>

<p><strong>CDA</strong> is the most successful HL7 v3 standard, widely used for clinical document exchange. CDA uses XML to structure medical documents, including:</p>

<ul>
<li><p><strong>Header</strong> — metadata (patient, author, institution, creation date)</p></li>
<li><p><strong>Body</strong> — clinical content, possible at 3 levels:
<ul>
<li><strong>Level 1</strong>: non-structured body (PDF/text)</li>
<li><strong>Level 2</strong>: sections with narrative text</li>
<li><strong>Level 3</strong>: fully structured, coded entries</li>
</ul></p></li>
</ul>

<p>CDA is widely used in <strong>C-CDA (Consolidated CDA)</strong> in the US for Meaningful Use/Promoting Interoperability, and in many projects in Europe and Japan.</p>

<h3 id="han-che-cda"><strong>Limitations of CDA</strong></h3>

<ul>
<li><p>Just fits <strong>document-based exchange</strong> (exchange documents)</p></li>
<li><p>Not supported <strong>data-level exchange</strong> (query each data field)</p></li>
<li><p>XML is complex, need to understand RIM</p></li>
<li><p>Does not support modern mobile/web apps</p></li>
</ul>

<h2 id="6-fhir-ra-doi"><strong>6. FHIR is born — the new "Fire" for interoperability</strong></h2>

<h3 id="nguon-goc-fhir"><strong>Origin</strong></h3>

<p>Year <strong>2011</strong>, <strong>Grahame Grieve</strong> — one of HL7's most veteran developers — proposes a completely new approach. Instead of trying to model everything (like v3), he suggested:</p>

<blockquote>
<p>"Build a set of simple, dynamically composable Resources, based on modern web technologies (REST, JSON, OAuth), and apply the 80/20 principle — solving 80% use-cases with 20% complexity."</p>
</blockquote>

<p>Name <strong>FHIR</strong> (pronounced "fire") is an abbreviation for <strong>Fast Healthcare Interoperability Resources</strong>, reflecting the goals:</p>

<ul>
<li><p><strong>Fast</strong> — quick to implement, easy to learn</p></li>
<li><p><strong>Healthcare</strong> — focuses on health</p></li>
<li><p><strong>Interoperability</strong> — interoperability between systems</p></li>
<li><p><strong>Resources</strong> — basic composable unit of data</p></li>
</ul>

<h3 id="cac-milestone-fhir"><strong>Development milestones of FHIR</strong></h3>

<table>
<thead>
<tr><th>Year</th><th>Version</th><th>Outstanding features</th></tr>
</thead>
<tbody>
<tr><td>2012</td><td>DSTU 0 (Draft)</td><td>First test version</td></tr>
<tr><td>2014</td><td>DSTU 1 (R1)</td><td>First Draft Standard for Trial Use</td></tr>
<tr><td>2015</td><td>DSTU 2 (R2)</td><td>Adoptions began to increase sharply</td></tr>
<tr><td>2017</td><td>STU 3 (R3)</td><td>Standard for Trial Use, many new Resources</td></tr>
<tr><td>2019</td><td>R4</td><td><strong>Normative first</strong> — Patient, Observation, Bundle stable</td></tr>
<tr><td>2020</td><td>R4B</td><td>Small update of R4</td></tr>
<tr><td>2023</td><td><strong>R5</strong></td><td>Current version — Topic-based subscriptions, many improvements</td></tr>
<tr><td>~2026+</td><td>R6</td><td>In development — AI/ML integration, improved Workflow</td></tr>
</tbody>
</table>

<h3 id="tai-sao-fhir-thanh-cong"><strong>Why is FHIR successful?</strong></h3>

<ol>
<li><p><strong>Based on web standards</strong> — REST, JSON, XML, OAuth 2.0, HTTP</p></li>
<li><p><strong>Easy to implement</strong> — many developers have interfaces running in 1 day</p></li>
<li><p><strong>Specifications are free</strong> — no license fee</p></li>
<li><p><strong>Many libraries support</strong> — HAPI FHIR (Java), fhir.js, fhirclient.py, Firely (.NET)</p></li>
<li><p><strong>Good extensibility</strong> — Extension mechanism allows extension without breaking the standard</p></li>
<li><p><strong>Human-readable</strong> — each Resource has an HTML narrative section</p></li>
<li><p><strong>Supports multiple paradigms</strong> — REST, Messaging, Documents, Services</p></li>
<li><p><strong>Government required</strong> — US (ONC/CMS), Australia, UK, EU all have mandate</p></li>
</ol>

<h2 id="8-fhir-tren-toan-cau"><strong>8. FHIR globally — Who is using it?</strong></h2>

<h3 id="hoa-ky"><strong>United States</strong></h3>
<ul>
<li><p><strong>21st Century Cures Act</strong> (2020): Require EHR vendors to support FHIR API (US Core)</p></li>
<li><p><strong>CMS Interoperability Rules</strong>: Require payers (insurance) to provide Patient Access API based on FHIR</p></li>
<li><p><strong>ONC TEFCA</strong>: National Data Exchange Framework, FHIR is the foundation</p></li>
<li><p>Epic, Cerner (Oracle Health), Allscripts all have FHIR APIs</p></li>
</ul>

<h3 id="chau-au"><strong>Europe</strong></h3>
<ul>
<li><p><strong>European Health Data Space (EHDS)</strong>: EU Regulation uses FHIR for cross-border health data</p></li>
<li><p><strong>International Patient Summary (IPS)</strong>: Based on FHIR, allows international sharing of clinical summaries</p></li>
</ul>

<h3 id="uc"><strong>Australia</strong></h3>
<ul>
<li><p><strong>AU Base Implementation Guide</strong>: Standard FHIR profile for the whole country</p></li>
<li><p><strong>My Health Record</strong>: National health records system using FHIR</p></li>
</ul>

<h3 id="viet-nam"><strong>Vietnam</strong></h3>
<ul>
<li><p>There is no official mandate on FHIR, but it is on the roadmap to digitize healthcare</p></li>
<li><p>Circular 54/2017/TT-BYT regulating medical data interoperability standards (FHIR not yet used)</p></li>
<li><p>Circular 46/2018/TT-BYT on electronic medical records</p></li>
<li><p>Several pioneering projects are testing FHIR</p></li>
<li><p>Great opportunity for building Vietnam FHIR Implementation Guide</p></li>
</ul>
