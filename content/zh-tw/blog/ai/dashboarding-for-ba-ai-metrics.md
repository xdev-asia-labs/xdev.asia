BA不需要成為資料分析師才能建立儀表板。有了Looker Studio或Power BI，只要你知道要追蹤什麼資料以及原因，幾小時內就能建立出實用的儀表板。

---

## 1. AI功能儀表板的指標框架

BA需要3層指標：

### 第一層：業務成果指標
「AI功能對業務有貢獻嗎？」

| 指標 | 說明 | 範例 |
|---|---|---|
| **自動化率** | AI自主處理的請求百分比 | 「AI自助解決73%的票務」 |
| **節省時間** | 與手動流程相比節省的時間 | 「每張票減少4.2分鐘」 |
| **導入AI後的錯誤率** | 導入AI後的錯誤率 | 「從12%下降至3.5%」 |
| **每次結果成本** | AI成本vs手動成本 | 「$0.03/票 vs $1.20手動」 |

### 第二層：AI品質指標
「AI運作正常嗎？」

| 指標 | 說明 | 警報門檻 |
|---|---|---|
| **精確度（7天滾動）** | 正確預測的百分比 | < 85% |
| **人工覆寫率** | 代理人覆寫AI的案例百分比 | > 25% |
| **信心分數分佈** | 信心分數的直方圖 | 10%以上的偏移 |
| **延遲P95** | 第95百分位的回應時間 | > 3秒 |

### 第三層：運營指標
「系統穩定嗎？」

| 指標 | 說明 | 警報 |
|---|---|---|
| **API錯誤率** | 返回錯誤的請求百分比 | > 1% |
| **佇列深度** | 等待人工審查的案例數 | > SLA容量 |
| **吞吐量** | 每小時請求數 | 從基準下降50%以上 |

## 2. 為不同利害關係人設計儀表板

### 2.1 管理層儀表板（月度審查）

**對象：** C層、總監  
**目的：** ROI和業務影響  
**內容：** 4到5個數字 + 1到2個趨勢圖

```
┌─────────────────────────────────────────────────────┐
│  AI功能健康儀表板 — [YYYY年MM月]                    │
├─────────────┬─────────────┬────────────┬────────────┤
│  處理的     │  AI自動     │  節省的    │  節省的    │
│  案例數     │  處理率     │  時間      │  成本      │
│  12,450件   │  73%        │  847小時   │  $34,200   │
│  (+8% MoM)  │  (+3% MoM)  │            │  vs手動    │
├─────────────┴─────────────┴────────────┴────────────┤
│  [趨勢圖：過去6個月的自動化率]                      │
└─────────────────────────────────────────────────────┘
```

### 2.2 運營儀表板（每日監控）

**對象：** BA＋運營團隊  
**目的：** 早期問題偵測  
**內容：** 即時指標、警報

主要小工具：
- 即時精確度（24小時滾動）
- 今日人工覆寫率 vs 上週
- 佇列深度 + SLA違反數
- 錯誤率 + 延遲

### 2.3 衝刺審查儀表板

**對象：** 開發團隊 + PM  
**目的：** 衝刺進度  
**內容：** 完成的故事 vs 計畫、發現的錯誤、AC覆蓋率

## 3. 工具比較

| 工具 | 優點 | 限制 | BA適合度 |
|---|---|---|---|
| **Looker Studio**（Google） | 免費、與Google試算表容易整合 | 視覺效果有限 | ✅ 最適合快速設定 |
| **Power BI** | 強大、與Office 365整合 | 付費、學習曲線陡峭 | ✅ 最適合企業 |
| **Metabase** | 支援SQL、自建主機選項 | 需要DB連線 | ⚠️ 設定需要技術支援 |
| **Tableau** | 最佳視覺效果 | 昂貴 | ❌ 對BA而言過於複雜 |
| **Grafana** | 運營/ML監控 | 過於技術性 | ❌ 不適合BA |

## 4. 步驟說明：BA的Looker Studio

### 步驟1：識別資料來源
請開發人員將日誌匯出到Google試算表或BigQuery：
```
所需欄位：
- timestamp
- request_id
- ai_prediction
- confidence_score
- human_override (true/false)
- override_reason
- resolution_time_seconds
```

### 步驟2：連接資料來源
Looker Studio → 新增資料 → Google試算表（或BigQuery）

### 步驟3：建立所需圖表

```
圖表1：計分卡 — 「自動化率」
  指標：COUNT（WHERE human_override = false）/ COUNT(*) × 100

圖表2：時間序列 — 「每日精確度（7天滾動）」
  維度：日期
  指標：正確預測的百分比

圖表3：長條圖 — 「最常見覆寫原因」
  維度：override_reason
  指標：COUNT(*)

圖表4：量規 — 「今日覆寫率」
  參考線：目標門檻
```

### 步驟4：設定自動更新
在Google試算表中：工具 → Apps Script → setTrigger（每日從DB更新）

## 5. 儀表板治理

BA應該建立的規則：

| 規則 | 說明 |
|---|---|
| **負責人** | 誰維護儀表板？（通常是BA） |
| **更新頻率** | 依指標而定：每日/每時/即時 |
| **發送** | 誰收到電子郵件摘要？（每週） |
| **審查頻率** | 每季度審查指標定義 |
| **警報路由** | 警報觸發時，誰收到以及應該做什麼？ |

## 結論

儀表板不是資料分析師的工作——而是**BA的溝通工具**。目標不是美觀或複雜，而是**適當的指標、適當的對象、適當的頻率**。每天都有人查看的5個圖表儀表板，比沒有人開啟的20個圖表儀表板更有價值。
